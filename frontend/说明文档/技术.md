### 一、 运筹优化与算法理论（核心必考）
**考察目的：** 验证你的数学底子，看你是不是只会调包（OR-Tools），还是真的懂原理。

1.  **精确算法 vs. 启发式算法 vs. 元启发式算法的区别？**
    *   **考点：** 什么时候用哪种？
    *   **策略：**
        *   **精确算法 (Exact)**：如分支定界 (Branch & Bound)。求全局最优，适合小规模问题（DREO 的简单装箱）。
        *   **启发式 (Heuristic)**：如贪心算法。速度快，解一般，适合超大规模实时响应。
        *   **元启发式 (Meta-heuristic)**：如遗传算法 (GA)、模拟退火 (SA)、禁忌搜索 (TS)。在搜索空间中跳出局部最优，适合复杂的 VRP/JSSP 问题。**（强调你在硕士论文里用过 TS）**

2.  **线性规划 (LP) 与 混合整数规划 (MIP) 的区别？**
    *   **考点：** 建模基础。
    *   **策略：** LP 变量是连续的（如倒水）；MIP 变量包含整数（如派车数量，你不能派 1.5 辆车）。MIP 求解难度是 NP-hard，通常用松弛变量或割平面法求解。

3.  **VRP 问题的经典求解思路有哪些？**
    *   **考点：** 你的老本行。
    *   **策略：** 提到 **列生成算法 (Column Generation)** 或 **大邻域搜索 (LNS/ALNS)**。OR-Tools 内部其实大量使用了 LNS。你可以顺带提一下你在代码里是如何配置 LNS 参数的。

4.  **Gurobi 和 OR-Tools (CP-SAT) 的底层原理有什么不同？**
    *   **考点：** 工具选型深度。
    *   **策略：** Gurobi 主要是基于单纯形法 + 分支定界（MIP Solver）；OR-Tools 的 CP-SAT 是基于**约束传播 (Constraint Propagation)** 和 **SAT 求解器**。CP 在处理“排程问题”（不重叠、累积资源）时往往比 MIP 更快。

### 二、 Python 编程与性能优化（高频）
**考察目的：** 既然你是“全栈”，且算法需要高性能，必须考察 Python 的底功。

1.  **Python 的 GIL (全局解释器锁) 是什么？对你的算法有什么影响？**
    *   **考点：** 为什么 Python 慢？怎么解决？
    *   **策略：**
        *   **解释：** GIL 导致同一时刻只能有一个线程在 CPU 上执行。
        *   **影响：** 对于 CPU 密集型任务（如大规模 VRP 求解），多线程没用。
        *   **解决：** 使用 **多进程 (Multiprocessing)** 或者调用 C/C++ 编写的底层库（如 NumPy, OR-Tools 本身就是 C++ 写的好在 Python 只是接口）。

2.  **生成器 (Generator) 和 迭代器 (Iterator) 的区别？**
    *   **考点：** 内存优化。
    *   **策略：** 比如处理 DREO 的百万级历史订单数据时，不能一次性 `read_csv` 进内存，要用 `yield` 关键字做生成器，流式处理数据，节省内存。

3.  **深拷贝 (Deepcopy) vs. 浅拷贝 (Copy)？**
    *   **考点：** 基础坑点。
    *   **策略：** 在做启发式算法（如遗传算法）时，复制解（Solution）必须用深拷贝，否则修改子代会影响父代。

4.  **FastAPI 的异步 (Async/Await) 是如何工作的？**
    *   **考点：** 你的 Web 框架知识。
    *   **策略：** 它是基于协程的。适合 I/O 密集型任务（如查数据库、请求外部 API）。但要注意，如果在 `async def` 里跑 OR-Tools 这种 CPU 密集型代码，会阻塞整个事件循环，必须用 `run_in_executor` 扔到线程池里跑。

**考察目的：** 供应链算法通常需要“销量预测”作为输入，考察你的知识广度。

2.  **LLM (大模型) 的 Function Calling 原理是什么？**
    *   **考点：** 针对你简历里的 LLM Agent。
    *   **策略：** LLM 不是直接执行函数，而是输出一个 JSON 格式的结构化数据（包含函数名和参数），然后由我们的 Python 后端解析这个 JSON，执行真正的代码（如 `solve_vrp`），最后把结果返还给 LLM 润色输出。

### 四、 数据库与 SQL（数据处理基础）
**考察目的：** 算法工程师大部分时间在洗数据。

1.  **索引 (Index) 的原理？B+ 树是什么？**
    *   **策略：** 加速查询。B+ 树减少磁盘 IO 次数。对于订单表，要在 `order_id`, `create_time` 上建索引。

2.  **SQL 优化：如何处理慢查询？**
    *   **策略：** `EXPLAIN` 查看执行计划；避免 `SELECT *`；避免在 `WHERE` 字段上做函数运算；大表分页优化等。

3.  **Redis 在你的系统中起了什么作用？**
    *   **策略：**
        *   **缓存：** 缓存 SKU 的基础信息（长宽高），减少 DB 压力。
        *   **消息队列：** 配合 Celery 做异步任务队列（简历里的“异步求解架构”）。

### 五、 系统架构与 Docker（工程化能力）
**考察目的：** 验证你是否具备独立交付项目的能力。

1.  **Docker 镜像分层是什么意思？**
    *   **策略：** 每一条指令（COPY, RUN）都是一层。修改代码时，利用缓存层可以加速构建。建议把 `pip install requirements.txt` 放在代码复制之前，利用缓存。

2.  **Celery 是怎么实现异步任务的？**
    *   **策略：** Producer（FastAPI）发送任务 -> Broker（Redis/RabbitMQ）存储消息 -> Worker（后台进程）监听并消费消息 -> Result Backend 存储结果。

3.  **WebSocket 断连了怎么处理？**
    *   **策略：** 你的前端看板用了 WebSocket。需要在这个层面上考虑 **心跳机制 (Heartbeat/Ping-Pong)** 和 **断线重连 (Reconnection)** 机制。
